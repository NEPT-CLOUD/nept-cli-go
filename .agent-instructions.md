# Developer & AI Agent Guidelines for `nept-cli-go`

Welcome! This codebase is designed to be highly modular, testable, and optimized for AI coding agents to inspect, run, and extend. Follow these guidelines to keep the repository clean and maintainable.

---

## Codebase Architecture

The project follows a standard Go command structure:
- **`cmd/`**: CLI command registrations (using Cobra).
  - `root.go`: Root command configuration.
  - `schema.go`: Introspection command that exports CLI specifications.
- **`internal/app/`**: Contain the unified `App` structure (context, configuration, logger, and mockable stdout/stderr streams).
- **`internal/config/`**: Configuration management using Viper.
- **`internal/logger/`**: Structured logging handler using standard `log/slog`.

---

## 1. Subcommand Implementation Rules

When creating a new subcommand (e.g., `cmd/foo.go`):

### A. Encapsulate Flags (No Package Globals)
**Do not** define variables for flags at the package level. They pollute scope and cause state-leakage between tests. Define flag variables locally within the command builder function closure:

```go
// GOOD
func NewFooCmd(appContainer *app.App) *cobra.Command {
    var (
        argValue string
        flagBool bool
    )

    cmd := &cobra.Command{
        Use:   "foo",
        Short: "Does foo task",
        RunE: func(cmd *cobra.Command, args []string) error {
            // Safe access to argValue and flagBool
            return nil
        },
    }

    cmd.Flags().StringVarP(&argValue, "arg", "a", "", "Description")
    cmd.Flags().BoolVarP(&flagBool, "flag", "f", false, "Description")

    return cmd
}
```

---

### B. Standardize Command Outputs
Never call `fmt.Println` or write manually to stdout/stderr inside subcommands. Use the unified `PrintResult` and `PrintErr` methods defined on `app.App`. This ensures both raw text and JSON outputs work automatically based on global flags (`--format text` or `--format json`):

```go
type FooResponse struct {
    Success bool   `json:"success"`
    Result  string `json:"result"`
}

// Inside RunE:
response := FooResponse{Success: true, Result: "completed"}
return appContainer.PrintResult("Task successfully completed!", response)
```

---

### C. Standardize Errors
Do not output error logs or format error strings manually. Return errors directly from `RunE`. The root runner catches and formats errors cleanly (in plain text or JSON):

```go
// Inside RunE:
if err != nil {
    // Return standard error to let root format it cleanly.
    return fmt.Errorf("failed to complete task: %w", err)
}
```

## 2. Command Discovery & Introspection

The CLI comes built-in with a command-schema introspection command:
```bash
go run main.go schema --format json
```

This dumps a recursive JSON structure of all registered subcommands, flags, shorthands, argument types, defaults, and requirements. AI agents should run this to understand all available CLI commands programmatically.

---

## 3. Testing Standards

- Create a test file alongside the command (e.g., `cmd/foo_test.go`).
- Use `app.App` to inject mock buffers (`bytes.Buffer`) for `Out` and `ErrOut`. This keeps command execution side-effect free.
- Run the full test suite using:
  ```bash
  go test -v ./...
  ```
